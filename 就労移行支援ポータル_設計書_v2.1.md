## 1. システム概要

### 1-1. 目的

就労移行支援事業所における利用者と管理者のコミュニケーション・申請業務をデジタル化し、業務効率を改善するポータルサイト。

### 1-2. 使用技術スタック

| カテゴリ | 技術・サービス |
|---|---|
| フロントエンド | HTML / CSS / Vanilla JavaScript |
| 認証 | Firebase Authentication（Google OAuth） |
| データベース | Cloud Firestore |
| ホスティング | Firebase Hosting |

### 1-3. ユーザー種別

| ロール | 説明 |
|---|---|
| user | 就労移行支援事業所の利用者。通所予定・面談申請を行う。 |
| admin | 事業所スタッフ（支援員）。申請の承認・面談確定を行う。 |

### 1-4. 設計上の制約・前提

- 利用者はGoogleアカウントでのみログイン可能
- 管理者アカウントはFirestoreへの直接書き込みで初期設定（管理者作成UIは本バージョンのスコープ外）
- タイムシートの印刷フォーマットは事業所指定の様式に準拠する（別途確認が必要）
- モバイル対応：スマートフォン（375px以上）での利用を想定。レスポンシブデザインを採用。
- **タイムシートおよびその通所予定はサブコレクションを使わず、すべて配列フィールドとして1ドキュメントに保持する。**

**loadNotifications() の取得ロジック**

以下の3コレクションに対して、`status` が `approved` または `rejected` かつ `reviewedAt` が `lastLoginAt` より新しいドキュメントを取得し、新着通知として表示する。

```
対象コレクション：
  timesheetRequests          where uid == currentUid, status in ["approved","rejected"], reviewedAt > lastLoginAt
  attendanceChangeRequests   同上
  interviewRequests          同上（rejected のみ。scheduled は interviews から取得）

表示件数：各10件・reviewedAt 降順
```

## 5. Firestoreコレクション設計

### 5-1. コレクション一覧

| コレクション名 | 用途 |
|---|---|
| users | ログインユーザー情報 |
| announcements | 管理者が投稿するお知らせ |
| timesheetRequests | 申請中・審査済みタイムシート申請 |
| timesheets | 承認済み月次通所予定（確定データ） |
| attendanceChangeRequests | 日単位の通所変更申請 |
| interviewRequests | 利用者の面談希望申請 |
| interviews | 管理者が確定した面談予定 |

> **設計方針：サブコレクションは使用しない。** 通所予定データはすべて配列フィールドとして親ドキュメントに保持する。1か月分は最大31件のためFirestoreの1MBドキュメント制限に十分収まり、1クエリで取得完結できる。

---

### 5-2. ① users

ドキュメントID：uid（Firebase AuthのUID）

| フィールド | 型 | 説明 |
|---|---|---|
| uid | string | Firebase AuthのUID |
| email | string | メールアドレス |
| username | string | 表示名 |
| role | string | `admin` / `user` |
| status | string | `active` / `inactive` |
| createdAt | Timestamp | アカウント作成日時 |
| lastLoginAt | Timestamp | 最終ログイン日時（通知表示の基準に使用） |

---

### 5-3. ② announcements

ドキュメントID：ランダム生成

| フィールド | 型 | 説明 |
|---|---|---|
| title | string | タイトル |
| body | string | 本文 |
| createdBy | string | 投稿者のuid |
| createdAt | Timestamp | 投稿日時 |
| updatedAt | Timestamp | 最終更新日時（編集時に更新） |

---

### 5-4. ③ timesheetRequests（申請中タイムシート）

ドキュメントID：ランダム生成

| フィールド | 型 | 説明 |
|---|---|---|
| uid | string | 申請者のuid |
| yearMonth | string | 対象年月（例: `"2026-04"`） |
| status | string | `pending` / `approved` / `rejected` |
| attendances | array | 通所予定の配列（下記参照） |
| rejectionComment | string | 却下コメント |
| createdAt | Timestamp | 申請日時 |
| reviewedAt | Timestamp | 承認・却下日時 |
| reviewedBy | string | 承認・却下した管理者のuid |

**attendances 配列の要素型**

```json
{
  "date": "2026-04-07",
  "startTime": "10:00",
  "endTime": "15:00"
}
```

> **重複申請防止：** 同一 uid + yearMonth で `status` が `pending` または `approved` のドキュメントが存在する場合、新規申請は不可とする。クライアント側で申請前に存在チェックを行い、重複時は `showToast("該当月は既に申請中または承認済みです", "error")` を表示して送信を中断する。

### 5-5. ④ timesheets（承認済みタイムシート）

## 6. セキュリティルール設計

### 6-1. 基本方針

- 認証済みユーザーのみFirestoreにアクセス可能
- **roleフィールドは管理者のみ変更可能。本人は変更不可。**
- 利用者は自分のデータのみ読み書き可能
- 管理者はすべてのデータを読み書き可能
- `status=inactive` のユーザーはクライアント側（auth.js）でログアウト処理を行い、セッションを終了させる

### 6-2. コレクション別ルール

| コレクション | read | create | update | delete |
|---|---|---|---|---|
| users（自分） | 本人・admin | 本人（初回登録のみ） | admin のみ | 不可 |
| announcements | 全認証ユーザー | admin | admin | admin |
| timesheetRequests | 本人・admin | 本人 | admin（承認・却下） | 不可 |
| timesheets | 本人・admin | admin（承認時） | admin | 不可 |
| attendanceChangeRequests | 本人・admin | 本人 | admin | 不可 |
| interviewRequests | 本人・admin | 本人 | admin・本人（取り下げのみ） | 不可 |
| interviews | 本人・admin | admin | admin | 不可 |

### 6-3. Firestore Security Rules

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function isAdmin() {
      return get(/databases/$(database)/documents/users/$(request.auth.uid)).data.role == 'admin';
    }

    function isOwner(uid) {
      return request.auth.uid == uid;
    }

    match /users/{uid} {
      allow read: if isOwner(uid) || isAdmin();
      allow create: if isOwner(uid);
      allow update: if isAdmin(); // role・statusの変更は管理者のみ
    }

    match /announcements/{id} {
      allow read: if request.auth != null;
      allow create, update, delete: if isAdmin();
    }

    match /timesheets/{id} {
      allow read: if isOwner(resource.data.uid) || isAdmin();
      allow create, update: if isAdmin();
    }

    match /timesheetRequests/{id} {
      allow read: if isOwner(resource.data.uid) || isAdmin();
      allow create: if request.auth.uid == request.resource.data.uid;
      allow update: if isAdmin();
    }

    match /attendanceChangeRequests/{id} {
      allow read: if isOwner(resource.data.uid) || isAdmin();
      allow create: if request.auth.uid == request.resource.data.uid;
      allow update: if isAdmin();
    }

    match /interviewRequests/{id} {
      allow read: if isOwner(resource.data.uid) || isAdmin();
      allow create: if request.auth.uid == request.resource.data.uid;
      // 本人はstatus=cancelled への更新（取り下げ）のみ許可
      allow update: if isAdmin()
        || (isOwner(resource.data.uid)
            && request.resource.data.status == 'cancelled'
            && request.resource.data.uid == resource.data.uid);
    }

    match /interviews/{id} {
      allow read: if isOwner(resource.data.uid) || isAdmin();
      allow create, update: if isAdmin();
    }

  }
}
```

## 8. データフロー設計

### 8-1. タイムシート承認フロー（最重要）

`approveTimesheetRequest(id, adminUid)` の内部処理：

1. `timesheetRequests/{id}` を取得（uid, yearMonth, attendances 配列）
2. `timesheets/{uid}_{yearMonth}` ドキュメントが既に存在するか確認
3. 存在する場合：既存ドキュメントを `attendances` 配列ごと上書き（`setDoc` with merge:false）
4. 存在しない場合：新規作成
5. `timesheetRequests/{id}` の status を `approved` に更新（reviewedAt, reviewedBy も更新）

### 8-2. 通所変更申請 承認フロー

`approveAttendanceChangeRequest(id, adminUid)` の内部処理：

1. `attendanceChangeRequests/{id}` を取得（requestType, date, newDate 等）
2. `date` の年月部分から `timesheetId = uid + "_" + yearMonth` を導出
3. `timesheets/{timesheetId}` を取得し、`attendances` 配列を取り出す
4. `requestType` ごとに配列を操作：
   - `create`：`attendances` に `{ date: newDate, startTime, endTime }` を追加
   - `cancel`：`attendances` から `date` が一致する要素を削除
   - `update`：`attendances` から `date` が一致する要素を削除 → `newDate` で新要素を追加
5. 操作後の配列で `timesheets/{timesheetId}` を `updateDoc`
6. `attendanceChangeRequests/{id}` の status を `approved` に更新

### 8-3. 面談確定フロー

`createInterview(requestId, date, startTime, endTime, staffId, adminUid)` の内部処理：

1. `interviews` コレクションに新規ドキュメントを作成（date, startTime, **endTime**, staffId を含む）
2. `interviewRequests/{requestId}` の status を `scheduled` に更新し、`interviewId` に作成したドキュメントのIDを保存

### 8-4. 面談キャンセルフロー

1. `interviews/{id}` の status を `cancelled` に更新、`cancelledAt` を記録
2. `interviews/{id}` の `interviewRequestId` を参照し、`interviewRequests/{interviewRequestId}` の status を `pending` に戻す（利用者が再申請できる状態にする）

## 10. 非機能要件

### 10-1. レスポンシブ対応

| ブレークポイント | 対応方針 |
|---|---|
| 375px〜767px（スマホ） | サイドバーをハンバーガーメニューに。カレンダーはコンパクト表示。モーダルは全画面表示。 |

### 10-2. エラー処理

- Firestoreの読み書きエラーはすべてtry-catchで捕捉し、`showToast()` でユーザーに通知
- ネットワークエラー時はリトライを促すメッセージを表示
- 認証エラー時はindex.htmlへリダイレクト

### 10-3. タイムシート印刷

- 印刷フォーマット：事業所指定の様式（実装前に様式を確認・確定すること）
- 実装方針：`window.print()` + 印刷専用CSS（`@media print`）でA4縦に最適化
- 管理者は対象利用者・対象月を選択してPDF保存または印刷が可能

### 10-4. セキュリティ

- クライアントサイドのロール判定はUI表示制御のみに使用
- データの保護はFirestoreセキュリティルールで実施（6章参照）
- APIキー・設定値はFirebase SDK経由のみ使用し、Gitに秘匿情報を含めない

### 10-5. 今後の拡張候補（スコープ外）

- プッシュ通知（Firebase Cloud Messaging）：申請の承認・却下をリアルタイム通知
- 管理者アカウント作成UI
- 祝日カレンダーAPI連携（`isWeekday()` の精度向上）
- 面談のオンライン（ビデオ通話リンク）連携
